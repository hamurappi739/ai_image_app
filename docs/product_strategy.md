# Product Strategy

Мобильное AI-приложение для генерации портретов и фотосессий.

**Стек:** Flutter, FastAPI, Supabase, Gemini API, RuStore Billing  
**Основной маркет:** RuStore  
**Аудитория:** массовая, в том числе **40–60+** — простой и понятный продукт без технического жаргона.

---

## 1. Product positioning

### Принцип

Приложение должно быть **простым для массовой аудитории**, включая пользователей 40–60+.

Пользователь покупает **понятный результат** (фото, фотосессию), а не абстрактную валюту.

### Язык в UI (запрещено / разрешено)

| Не использовать в UI | Использовать вместо |
|----------------------|---------------------|
| tokens | — |
| credits | **генерации** |
| paid credits | **осталось генераций**, **пакет генераций** |
| prompt (как термин) | **идея**, **описание**, **что хотите получить** |

| Разрешённые формулировки |
|--------------------------|
| бесплатные генерации |
| генерации |
| фотосессия |
| пакет генераций |
| осталось генераций |

> **Технически** backend и БД могут хранить `paid_credits`, `credit_transactions` — это внутренняя модель. **Flutter** переводит её на понятный язык для пользователя.

## 2. Main monetization model

### Основная модель (простая)

**Платные фотосессии по фиксированной цене** — главный monetization path для массового пользователя.

| Что покупает пользователь | Пример |
|---------------------------|--------|
| Готовая **фотосессия** из 3 фото | **100 ₽** за одну фотосессию |

### Что такое фотосессия

- **3 фото** одного человека
- **Одна тематика** и **похожая локация**
- **Разные ракурсы**, единая атмосфера (ощущение одной съёмки)

### Воронка оплаты (критично)

1. Пользователь видит **примеры результата** (preview шаблона)
2. **Оплачивает** фотосессию
3. **Только после оплаты** backend запускает генерацию

**Платные фотосессии не генерируются до оплаты** — никакого «списания кредита втёмную» до покупки понятного пакета.

> Цены и точное число фото в сессии — гипотеза, уточняются после тестов.

## 3. Photoshoot templates

Раздел **«Фотосессии»** (в навигации — вкладка **Templates**).

### Каталог

- **10–15** готовых вариантов фонов / тем (динамический каталог позже)
- У каждого шаблона: preview, название, скрытый технический prompt

### MVP-политика доступа

| Тип | Поведение |
|-----|-----------|
| **2–3 бесплатных** шаблона | Можно создать после загрузки фото (когда upload будет готов) |
| **Остальные платные** | Показывают примеры, **требуют оплату перед генерацией** |

### Примеры будущих фотосессий

- Business portrait  
- Luxury portrait  
- Cozy home portrait  
- Winter photoshoot  
- City portrait  
- Studio portrait  
- Elegant evening portrait  
- Travel portrait

## 4. Custom prompt mode

Отдельный режим для **продвинутых** пользователей (вкладка **Create** / «Свой запрос»).

| Правило | Значение |
|---------|----------|
| Ввод | Свой текстовый запрос (без слова «промпт» в UI) |
| Бесплатный лимит | **3 генерации** (`FREE_GENERATIONS_LIMIT=3` на backend) |
| После лимита | Покупка **пакета генераций** |

Соответствует текущей backend-логике: `free_generations_used` → затем списание из `paid_credits` (1 генерация = 1 единица баланса).

## 5. Generation packages

**Дополнительная** опция для продвинутых пользователей (вкладка **Credits** / «Пакеты»).

В UI — только **«генерации»**, не кредиты и не токены. Пользователь должен сразу понимать: **сколько изображений** сможет создать.

### Примерные пакеты (гипотеза)

| Пакет | Генераций | Цена |
|-------|-----------|------|
| Starter | 25 | **199 ₽** |
| Creator | 100 | **499 ₽** |
| Pro | 250 | **1199 ₽** |

> Цены не финальные. Подписки в MVP **не планируются**.

### Связь с backend

Технически: пополнение `paid_credits`, списание **1** за одну custom-генерацию. В UI: «осталось **N** генераций».

## 7. Backend implications

### Текущая credits-логика (оставить технически)

| Сущность | Назначение |
|----------|------------|
| `free_generations_used` | Счётчик бесплатных custom-генераций |
| `paid_credits` | Баланс для custom mode и совместимости с пакетами |
| `credit_transactions` | Аудит начислений и списаний |
| `FREE_GENERATIONS_LIMIT` | Лимит free (MVP: 3) |

### Отображение в UI

| Backend | UI |
|---------|-----|
| `paid_credits` | Осталось **N** генераций |
| free limit | Бесплатные генерации: **X** из **Y** |
| purchase | **Пакет генераций** |

### Будущие сущности для фотосессий

| Сущность | Поля / смысл |
|----------|----------------|
| `photoshoot_templates` | Каталог тем, preview, hidden prompt |
| `photoshoot_orders` | Заказ после оплаты |
| `price_rub` | Фиксированная цена (напр. 100 ₽) |
| `photos_count` | 3 |
| `is_free` | Бесплатный шаблон в MVP |
| `requires_payment` | Генерация только после оплаты |

Алгоритм custom-генерации (кратко):

1. `free_generations_used < FREE_GENERATIONS_LIMIT` → free, +1 used  
2. Иначе `paid_credits > 0` → paid, −1 credit  
3. Иначе → ошибка «нет генераций», экран покупки  

См. [database_schema.md](database_schema.md).

## 9. MVP decision

### Ближайший MVP

| Делаем | Не делаем |
|--------|-----------|
| Текущий **custom prompt flow** (Create + `POST /generate`) | Реальные платежи RuStore |
| Вкладка **Templates** как **photoshoots** | Генерация платной фотосессии до оплаты |
| **2–3** бесплатных photoshoot templates (UI + mock) | Полная загрузка фото пользователя |
| Платные photoshoot templates: UI **«100 ₽, оплата будет добавлена позже»** | Watermark / preview-only flow |
| Вкладка **Credits** — UI пакетов генераций (заглушка) | Подписки |

### После MVP

- RuStore Billing: фотосессии **100 ₽** и пакеты генераций  
- Auth + синхронизация баланса  
- Реальная генерация 3 фото по `group_id`  
- RLS, webhook idempotency

## 10. Notes from competitor audit

- Конкуренты перегружают **токенами**, **кредитами**, подписками — отталкивает аудиторию 40+.
- **Фотосессия за фиксированную цену** проще объяснить, чем «купить 100 кредитов».
- **Бесплатные** шаблоны и 3 free custom-генерации — вход в продукт без оплаты.
- Готовые **фотосессии** снижают порог: не нужно придумывать «промпт».
