# App Design Strategy

Концепция дизайна и структуры **AI Image Generator** для мобильного приложения (Flutter).

**Аудитория:** **обычные пользователи**, в том числе **40–60+** — без технического бэкграунда; нужен **простой, понятный, «дорогой»** интерфейс с **крупными действиями** и короткими текстами.  
**Идея продукта:** не «писать промпты», а **выбрать готовую фотосессию** или **описать идею своими словами** (текст и/или фото).  
**Язык UI (текущий MVP):** основной пользовательский интерфейс — **русский** (вкладки, кнопки, подсказки, empty state).

---

## 1. Design direction

| Принцип | Реализация |
|---------|------------|
| Стиль | Светлый **премиальный** UI: фон `#F7F8FC`, белые **мягкие карточки**, лёгкие тени |
| Акцент | **Фиолетово-синий** (`#5B6CFF`, градиенты `#7C5CFF` → `#4A7CFF`) на CTA и выделениях |
| Карточки | Крупные preview-блоки (фотосессии, пакеты), удобный tap-target |
| Тексты | Короткие формулировки для **обычной аудитории 40–60+**, **минимум** технических терминов |
| Действия | **Крупные** понятные кнопки: «Далее», «Пропустить», «Создать изображение», «Попробовать» |
| Запрет в UI | **prompt**, **«промпт»**, **«кредиты»**, **«токены»**, **credits**, **tokens** |

### UX language rules

**Не использовать в видимом UI:** prompt, промпт, кредиты, токены, credits, tokens.

**Использовать:**

- **описание** (поле ввода на «Создать»)
- **идея** (быстрые chips «Попробуйте идею»)
- **изображение** / **изображений** (пакеты, галерея, результат)
- **бесплатные генерации** (*«3 бесплатные генерации»*, *«осталось: N бесплатных генераций»*)
- **пакеты**, **осталось** (изображений и фотосессий)
- **фотосессия** / **фотосессии**

**Не использовать «кредиты»** даже для внутреннего `paid_credits` в UI.

Технические `prompt`, `paid_credits`, `credit_consumed` — только backend / JSON (см. [api_contract.md](api_contract.md)).

## 2. First-run onboarding и контекстная помощь

### UX-подход

- **Общая обучалка** при **первом запуске** — знакомство с приложением целиком.
- **Контекстная помощь** в **отдельных разделах** — подсказки по текущей вкладке.
- Повторная помощь: кнопка **«Помощь»** в правом верхнем углу шапки раздела (текстовая, не иконка **?**).
- Тексты **короткие и простые**, для **обычных пользователей 40–60+**.
- В видимом UI **нельзя** использовать **prompt**, **tokens**, **credits** (и русские «промпт», «токены», «кредиты»).

### First-run onboarding (реализовано)

При **первом запуске** — **обучалка на 5 экранов** (`onboarding_completed` в локальном хранилище). После **«Начать»** или **«Пропустить»** **больше не показывается**.

**Формат:**

- Панель ~**половина / основная часть** экрана; крупный заголовок, короткий текст.
- На каждом экране: **«Далее»** и **«Пропустить»**; на последнем — **«Начать»**.
- Индикатор **«N из 5»** и точки.

**Экраны:**

| № | Тема | Смысл |
|---|------|--------|
| 1 | **Добро пожаловать** | Изображения и **фотосессии** из идей и фото |
| 2 | **«Создать»** | **Идея** словами → одно изображение |
| 3 | **«Фотосессии»** | Стиль + фото → несколько снимков в одном образе |
| 4 | **Хорошее исходное фото** | Лицо видно; размытые/тёмные фото хуже |
| 5 | **«Галерея»** | Все результаты; фотосессии — **одной карточкой**; **если забудете — нажмите «Помощь» в правом верхнем углу раздела** |

### Контекстная помощь по вкладкам (реализовано)

| Вкладка | Когда показывается | Содержание (кратко) |
|---------|-------------------|---------------------|
| **Создать** | Авто при **первом** открытии вкладки + **«Помощь»** всегда | Готовые идеи, описание, **фото для образа** (UI; backend позже), Галерея |
| **Фотосессии** | Авто при **первом** открытии вкладки + **«Помощь»** всегда | Стиль, хорошее фото, результат, Галерея, **«Своя фотосессия»** (UI; backend позже) |
| **Пакеты** | Без автопоказа | **«Помощь»** — смешанные пакеты, «Создать» vs фотосессии, «Своя сумма», оплата позже |

Диалог: **4 шага** (или список пунктов для **Пакетов**), **«Далее»** / **«Понятно»**; флаги `create_help_seen`, `photoshoots_help_seen`.

## 2.1. Качество генераций (требования руководства)

Генерации должны выглядеть **красиво и аккуратно**. Особенно избегать:

- кривых / искажённых лиц
- лишних людей в кадре
- коллажей, сеток, contact sheet
- низкого или «странного» качества

**Backend (план):** усилить **instruction** для Gemini (не показывать пользователю как «промпт»):

| Требование | Обычная генерация | Фотосессия |
|------------|-------------------|------------|
| realistic high-quality image | ✅ | ✅ |
| natural face | ✅ | ✅ |
| no distorted face | ✅ | ✅ |
| no extra faces | ✅ | ✅ |
| no collage / grid | ✅ | ✅ (уже в product) |
| one final image | ✅ по умолчанию | ❌ — **3 отдельных** файла, каждый = одна фото |

**Фотосессии:** каждый результат — **отдельная standalone-фотография**, не коллаж (реализовано в backend product rules; при доработке prompts — сохранить).

## 2.2. Долгие операции: явное состояние ожидания (реализовано)

**UX-принцип:** любая **долгая операция** (генерация изображения, подготовка фотосессии) должна **явно показывать**, что приложение **работает**. Пользователь не должен думать, что экран **завис**.

| Принцип | Реализация |
|---------|------------|
| Видимый прогресс | Блокирующее **модальное окно** по центру + **`CircularProgressIndicator`** |
| Ожидаемое время | **«Создать»** — до **~минуты** (обратный отсчёт **60 → 0** сек.); **«Фотосессии»** (запрос на backend) — до **~двух минут** (**120 → 0** сек.) |
| После нуля таймера | Если запрос ещё идёт: *«Почти готово, ждём результат...»* — окно **остаётся**, пока не придёт ответ |
| Ранний успех / ошибка | Окно **закрывается сразу**; при ошибке — **существующее** мягкое сообщение |
| Защита от случайных действий | **Фон затемнён**; закрытие по тапу вне окна и «назад» **отключены** на время генерации |
| Исключения | **Платные** фотосессии, **«Своя фотосессия»**, фото на **«Создать»** без backend — **без** этого modal |

Виджет: `GenerationProgressDialog` — параметры `title`, `subtitle`, `totalSeconds`. См. [project_status.md](project_status.md) § **«Ожидание генерации»**.

## 3. Main user idea

Пользователь видит **красивый пример** и хочет **похожее фото себя**.

### Главный сценарий (целевой)

1. Выбрать **шаблон** (портрет или фотосессию)
2. **Загрузить своё фото** (лицо / портрет)
3. Нажать **«Создать»**
4. Получить **результат**

**MVP:** загрузка фото — **mock / placeholder** (иконка «добавить фото», без реального upload). Реальная загрузка и face-swap — позже.

## 4. Bottom navigation (реализовано во Flutter)

Нижнее меню — **5 вкладок**, подписи на **русском**:

| Вкладка | EN (код) | Текущее состояние UI | Backend |
|---------|----------|----------------------|---------|
| **Создать** | Create | **Описание** + быстрые идеи + **UI-каркас фото** (picker/preview) → одно изображение по тексту (`POST /generate`); фото+описание — backend позже | `POST /generate` (текст); фото — **не отправляется** |
| **Фотосессии** | Photoshoots | **Каталог** 8 стилей + **«Своя фотосессия»** (UI-каркас) → bottom sheet / dialog; готовые: multipart upload | `POST /photoshoots/generate` (готовые стили); custom — **не подключено** |
| **Галерея** | Gallery | `GET /generations` + группировка фотосессий по `photoshoot_id`; empty state | `GET /generations` (dev: `TEST_USER_ID`) |
| **Пакеты** | Packs | Смешанные **199/499/999 ₽**, переключатель, **«Своя сумма»**, **«Помощь»**; оплата — placeholder | Нет оплаты / нет balance API |
| **Профиль** | Profile | Placeholder входа и будущих настроек | Нет auth |

`IndexedStack` сохраняет состояние вкладок при переключении.

**Навигация между «Создать» и «Галерея»:**

- **Галерея** (пусто) → «Создать первое изображение» → вкладка **Создать**
- **Создать** (после успешной генерации) → **«Открыть в Галерее»** → вкладка **Галерея**

Список истории в `MainShell` (`List<GeneratedImageItem>`): при старте — из backend, затем дополняется локально сверху.

## 6. Photoshoot mode (вкладка «Фотосессии»)

Раздел **«Фотосессии»** во Flutter — **главный простой сценарий** для массовой аудитории: выбрать готовый стиль без составления длинного описания.

### Ощущение продукта

Одна **тема**, одна **локация**, одна **атмосфера** — **3 отдельных файла** (3 standalone photos) в одном стиле одного человека с **разных ракурсов**, как от одного фотографа.

**Продуктовое требование к результату:** каждый generated image — **одна самостоятельная фотография**, не коллаж и не сетка. Запрещено в одном файле: collage, grid, contact sheet, split-screen, before/after, несколько панелей, несколько кадров внутри одного изображения. Фотосессия из **3 результатов** = **3 отдельных image file**, а не одна картинка-коллаж. Backend Gemini prompt это явно требует; при `PHOTOSHOOT_OUTPUT_COUNT=3` — **3 отдельных API-вызова**, каждый за одним standalone photo.

**Пример:** «Фотосессия в поле с пионами» — три кадра: анфас, три четверти, по пояс; одинаковый свет и фон.

### MVP credit hypothesis (не финально)

| Действие | Генерации (гипотеза) |
|----------|----------------------|
| Обычная генерация (1 фото) | 1 |
| Фотосессия (3 фото) | 3 |
| Бесплатный preview с watermark | 1 раз (на пользователя) |
| Скачать без watermark | после оплаты / пакета |

> Цены и экономика — **уточняются** после тестов (см. [product_strategy.md](product_strategy.md)). В UI не писать «кредиты».

**Текущий Flutter UI (MVP):**

- **8 готовых фотосессий** — основной **простой сценарий** для массовой аудитории (не нужно придумывать описание).
- **Каталог карточек (реализовано):** каждая карточка работает как элемент **каталога готовых образов** — gradient **placeholder preview** (место под будущий пример), **название**, **«3 фото»**, **«Бесплатно»** / **«100 ₽»**, **короткое описание**; платные мягко выделены (рамка/акцент).
- **Responsive layout:** 2 колонки на широком web, 1 колонка на узком экране.
- Кнопки **«Попробовать»** (бесплатные) / **«Оплата позже»** (платные) открывают **modal bottom sheet**.
- Текущий UX-цепочка: **выбрать стиль → рекомендации по фото → выбрать фото → preview → отправка на backend (бесплатно) → результат в Галерею или placeholder-сообщение**.

### UX-принцип: объяснять требования к фото заранее

**Лучше заранее объяснить**, какое исходное фото подходит, **чем показывать разочарование** после плохого результата. Поэтому в bottom sheet есть заметный блок **«Какое фото лучше загрузить»** — до выбора файла и до генерации.

### Карточки фотосессий — каталог (реализовано)

| Элемент | Сейчас |
|---------|--------|
| **Preview / пример** | Gradient **placeholder** + метка «Пример»; **реальные curated visuals — позже** |
| **Название** | «Студийный портрет» и т.д. |
| **Цена** | **«Бесплатно»** или **«100 ₽»** |
| **Объём** | **«3 фото»** |
| **Короткое описание** | 1–2 строки атмосферы стиля |

Placeholder examples are **temporary** and will be **replaced with curated visuals later**.

### Внутри выбранной фотосессии — рекомендации и примеры (реализовано)

В **bottom sheet**:

**«Какое фото лучше загрузить»** — простые пункты:

- **Лицо хорошо видно**
- Фото **не размытое**
- **Хорошее освещение**
- **Без сильной тени на лице**
- Для **некоторых образов** лучше фото **по пояс** или **в полный рост**
- **Чем лучше исходное фото, тем лучше результат**

**«Пример результата»** — **3 placeholder-миниатюры** («Фото 1», «Фото 2», «Фото 3»); не реальные изображения. Позже — curated examples.

**Позже:** заменить placeholders на **настоящие curated example images**; улучшить **visual branding / art direction**.

### Своя фотосессия — готовые стили vs свой образ

| | **Готовые фотосессии** | **«Своя фотосессия»** |
|---|------------------------|------------------------|
| Выбор | **Быстрый** — стиль из каталога | Пользователь **сам описывает** желаемый образ |
| Ввод | Стиль + фото | **Фото** + **свои пожелания** (текст) |
| Результат | Обычно **3** изображения в одном стиле | **Набор изображений** (как у готовых стилей) |

**Помощник «Как описать лучше»** — пользователь должен видеть подсказки про:

- **где** проходит съёмка (офис, улица, студия, природа)
- **образ** (деловой, спокойный, праздничный, уверенный)
- **одежду** (костюм, платье, casual)
- **фон** и **настроение**
- чем **понятнее описание**, тем лучше результат

### «Своя фотосессия» — UI-каркас (реализовано)

На вкладке **«Фотосессии»** — карточка **«Своя фотосессия»** (**«Скоро»**, **«3 фото»**) → **dialog**:

1. **«Добавить фото»** → preview → **«Убрать фото»**
2. Поле **«Ваши пожелания»** (multiline), hint с примером деловой фотосессии
3. Блок **«Как описать лучше»** (съёмка, образ, одежда, фон, настроение)
4. **«Создать позже»** → SnackBar *«Своя фотосессия будет добавлена позже»*

**Статус:** только **UI**; **backend / Gemini / Gallery / оплата — не подключены**.

### «Своя фотосессия» — backend (план)

1. Endpoint для upload + пользовательского текста.
2. Gemini → **набор изображений** → **`generations`** / **Галерея** (с **`photoshoot_id`**).
3. При необходимости — **оплата** (отдельное решение product owner).

**Пример пожеланий:** «Деловая фотосессия в светлом офисе, уверенный образ, спокойный фон.»

### Demo-экран выбора фото (bottom sheet)

Открывается по нажатию на карточку. **Сейчас — выбор фото локально + multipart upload на backend для бесплатного сценария**; при включённой генерации результат попадает в **Галерею**.

**Показывает:**

| Блок | Содержание |
|------|------------|
| Заголовок | Название фотосессии + **«3 фото»** + badge **«Бесплатно»** или **«100 ₽»** + описание |
| **Пример результата** | 3 **placeholder-миниатюры** («Фото 1–3»); позже — curated images |
| **Какое фото лучше загрузить** | Рекомендации по качеству исходного фото (лицо, резкость, свет, тень, пояс/рост) |
| Сценарий | «Загрузите своё фото…» → **3 изображения в выбранном стиле** |
| Выбор фото | Кнопка/блок «Выберите своё фото», выбор из галереи/файлов, preview выбранного фото и «Выбрать другое фото» |
| **Что получится** | 3 изображения в одном стиле; мягкая обработка лица и света; **результат появится в Галерее** |

**Кнопки:**

- **«Понятно»** — закрывает sheet.
- **«Подготовить позже»** / **«Оплата позже»**:
  - если фото не выбрано: мягкое сообщение **«Сначала выберите фото»**;
  - если фото выбрано и стиль **бесплатный**:
    - отправка `multipart/form-data` на `POST /photoshoots/generate`;
    - **генерация выключена** (backend `ENABLE_PHOTOSHOOT_GENERATION=false`, по умолчанию): **«Обработка фото будет добавлена позже»**;
    - **генерация включена** (`ENABLE_PHOTOSHOOT_GENERATION=true`, controlled test): backend → **`image_urls`** → **Галерея** (после reload — **одна grouped card** по `photoshoot_id`) → SnackBar **«Фотосессия готова»** → вкладка **Галерея**;
  - если фото выбрано и стиль **платный**: **«Оплата будет добавлена позже»** (backend upload пока не вызывается).

Технический статус **`501`**, HTTP-коды, backend `detail` и пути файлов пользователю **не показываются**.

В пользовательском UI **не** использовать «промпт», «кредиты», «токены».

**Позже (не в demo):** оплата 100 ₽ для платных стилей; **detail view** фотосессии в Галерее (опционально).

### Галерея: фотосессии — текущий UX

**Сейчас:**
- **1 фотосессия = 1 карточка** в Галерее (группировка по **`photoshoot_id`** из **`GET /generations`**).
- Внутри карточки — **мини-сетка изображений** этой фотосессии (1 большое; 2 рядом; 3 — одно сверху и два снизу).
- Подпись: **«Создано по описанию:»**, название (например **«Фотосессия: Домашний портрет»**), **«N изображений»**, дата/время.
- **Обычные изображения** (`/generate`, legacy rows с **`photoshoot_id=null`**) — **отдельные карточки**, как раньше.

**Почему так:** для аудитории **40–60+** проще видеть **одну связанную фотосессию**, чем **3 отдельных результата** с одинаковым описанием без явной связи.

**Позже (опционально):** tap по карточке → **экран деталей** с полноразмерным просмотром всех кадров фотосессии.

Примеры карточек: Студийный / Деловой / Домашний портрет (free); Премиум, Зимняя, Городской, Вечерний, Портрет в путешествии (100 ₽).

## 7. Gallery (Галерея)

Раздел **«Галерея»** — история созданных изображений: **загрузка с backend** при старте + **мгновенное** добавление новых результатов текущей сессии.

### Загрузка и данные (MVP)

| Аспект | Поведение |
|--------|-----------|
| При старте | `ApiService.fetchGenerations()` → **`GET /generations`** (limit по умолчанию 20) |
| Состояние | In-memory в `MainShell` (`GeneratedImageItem` + группировка в `GalleryDisplayItem` по `photoshootId`) |
| Новые за сессию | После **`POST /generate`** или **фотосессии** — элемент(ы) **сразу сверху**; фотосессия с backend **`photoshoot_id`** после reload = **одна карточка-группа** |
| Backend недоступен | Приложение **не падает**; ошибка не показывается SnackBar; в пустой галерее — мягкий текст (без технических деталей) |
| Debug-записи | В UI **скрываются** ответы с служебным описанием (`debug test prompt`, `debug`, `test prompt` — без учёта регистра); данные в Supabase **не удаляются** |
| **Очистить** | Кнопка в шапке галереи: сбрасывает **только** in-memory список на **текущем устройстве** / в текущей сессии; **не** удаляет строки в Supabase и **не** вызывает DELETE на backend |
| После очистки | Empty state; SnackBar «Галерея очищена на этом устройстве» (без формулировок «удалено навсегда») |
| Перезапуск приложения | При следующем старте **`GET /generations`** может **снова подтянуть** историю с backend (если записи есть в БД) |
| Аккаунт | Полноценная история **по авторизованному пользователю** — позже (сейчас dev: `TEST_USER_ID` на backend) |

### UI

**Если список пуст** (в т.ч. после фильтрации debug):

- Empty state: «Пока нет изображений», CTA **«Создать первое изображение»** → **Создать**
- Блок **«Скоро»**: полная история после входа в аккаунт

**Если есть элементы:**

- Subtitle: **«Ваши созданные изображения»**
- Справа в шапке: **«Очистить»** (`delete_outline`) — только локальный список (см. таблицу выше)
- Responsive-сетка (1 / 2 / 3 колонки): preview (одиночное или **мини-сетка фотосессии**), «Создано по описанию:» (не «промпт»), описание, «N изображений» для групп, время

**Позже:** detail view фотосессии; история только своего аккаунта; **удаление изображений из аккаунта/backend**; скачать, «создать похожее», фильтры.

## 8. Packs (Пакеты)

Вкладка **«Пакеты»** — покупка **смешанного баланса**: **обычные изображения** (вкладка **«Создать»**) и **фотосессии** (готовые стили / будущая «Своя»). В UI **не** использовать **credits**, **tokens**, **prompt**, **«кредиты»**, **«токены»**, **«промпт»**.

### Базовая экономика (product owner)

| Единица | Условная цена | Что получает пользователь |
|---------|---------------|---------------------------|
| **Обычное изображение** | **10 ₽** | **1** картинка по описанию (**«Создать»**) |
| **Фотосессия** | **100 ₽** | **3** готовых изображения в одном стиле |

Пакеты **объединяют** оба типа — фотосессии **не** продаются только отдельно от пакетов.

### Готовые пакеты — «С фотосессиями» (предварительно, цифры можно менять)

Логика: сумма пакета → сначала **фотосессии** по **100 ₽**, остаток → **обычные изображения** по **10 ₽**.

| Цена | Фотосессии | Обычные изображения |
|------|------------|---------------------|
| **199 ₽** | **1** | **9** |
| **499 ₽** | **3** | **19** |
| **999 ₽** | **8** | **19** |

### Альтернатива — «Только изображения»

Переключатель в UI: **«С фотосессиями»** / **«Только изображения»**.

| Цена | Только изображения |
|------|-------------------|
| **199 ₽** | **19** |
| **499 ₽** | **49** |
| **999 ₽** | **99** |

### Режим «Своя сумма» (реализовано во Flutter, без оплаты)

- Поле суммы: **сейчас** мин. **200 ₽** в UI; **план (руководство):** мин. **10 ₽** (1 изображение = 10 ₽), **макс. 100 000 ₽**.
- При **10 ₽** и **0** фотосессий → **1** изображение; фотосессия по-прежнему **100 ₽**.
- **Stepper / slider** — сколько фотосессий включить (не больше, чем позволяет сумма).
- Live-блок: *«За N ₽ вы получите:»* → фотосессии + изображения (остаток ÷ 10 ₽).
- **«Оплата скоро»** → SnackBar *«Оплата будет добавлена позже»*; при невалидной сумме — *«Введите сумму от 200 ₽»*.

**Пример 1000 ₽, 8 фотосессий:** 8 фотосессий + 20 изображений. **0 фотосессий:** 100 изображений.

### Текущий Flutter UI (реализовано)

| Элемент | Поведение |
|---------|-----------|
| Переключатель | **«С фотосессиями»** / **«Только изображения»** |
| Готовые пакеты | **199 / 499 / 999 ₽**; на карточке — цена, бейджи фотосессий/изображений, подпись |
| **«Популярный»** | Плашка на пакете **499 ₽** |
| Сетка | 3 колонки (web), 2 (tablet), 1 (mobile); адаптивная высота карточек |
| Помощь | **«Помощь»** → `PacksHelpDialog` |
| Оплата | **«Оплата скоро»** — placeholder, без RuStore |

### UX баланса и бесплатных генераций (план)

**Принцип:** в видимом UI **не** «кредиты», **credits**, **tokens**, **prompt**.

**Старт (новые пользователи):**

- Показать: *«Вам доступно 3 бесплатные генерации в разделе «Создать»»*.
- Или: *«Осталось: 3 бесплатные генерации»*.

**После покупок / пополнения:**

- *«Осталось: 12 изображений и 2 фотосессии»* (или раздельные строки — product owner).

| Место | Формат | Приоритет |
|-------|--------|-----------|
| **Профиль** | Основной баланс (бесплатные + платные) | **главный** |
| **Пакеты** (сверху) | Краткий остаток перед пакетами | **вторичный** |
| **Создать** | Бесплатные остаток + подсказка пополнить | **план** |
| **Фотосессии** | Остаток фотосессий / «нужно пополнить» | **позже** |

**Правила генерации:**

| Сценарий | Условие |
|----------|---------|
| **«Создать»** (обычная) | Сначала **3 бесплатные**; затем только при **платном** балансе изображений |
| **Платная фотосессия** | Только после **оплаты** или при **фотосессиях на балансе** |
| **Бесплатная фотосессия** | Как сейчас (без оплаты), в рамках каталога |

**Backend** должен **защищать** платные сценарии; **frontend** — объяснять: *«Пополните баланс в разделе «Пакеты»»* (без технических кодов).

**Что должно быть понятно пользователю 40–60+:** что покупает, сколько фотосессий и изображений получит, **сколько осталось** после использования.

**Следующий порядок:** см. [roadmap.md](roadmap.md) — мин. **10 ₽** → бесплатные 3 → баланс UI → идеи/подсказки → prompts → balance API → RuStore.

## 10. Create (Создать)

### Ожидание генерации (реализовано)

При нажатии **«Создать изображение»** (запрос на backend по описанию) — **блокирующее модальное окно** (см. § **2.2**):

- Заголовок: **«Создаём изображение»**
- Текст: **«Обычно это занимает до минуты.»**
- Обратный отсчёт: **«Осталось примерно: N сек.»** (**60 → 0**)
- После **0** при долгом ответе: **«Почти готово, ждём результат...»**

На **«Фотосессиях»** при бесплатной отправке на backend — **120 → 0** сек., заголовок **«Готовим фотосессию»**, подзаголовок про **до двух минут**. Платные карточки и **«Своя фотосессия»** — modal **не показывается**.

### Готовые идеи по категориям (план)

**Сейчас:** один общий ряд chips (Киберпанк-кот, Уютный дом, …).

**План:** разбить по **блокам/категориям**, чтобы быстрее найти идею:

| Категория | Примеры направлений |
|-----------|---------------------|
| **Природа** | пейзажи, лес, море |
| **Город** | улицы, архитектура, ночной город |
| **Люди / портрет** | портрет, деловой образ |
| **Дом / интерьер** | уютный интерьер, кухня |
| **Праздник** | праздничная атмосфера |
| **Реклама / товар** | предмет, рекламный кадр |
| **Соцсети / аватар** | аватар, обложка |

Tap по chip — вставка текста в поле описания (как сейчас).

### Блок «Как получить хороший результат» (расширение, план)

**Сейчас:** краткие подсказки + один пример.

**План:** больше советов + **несколько примерных описаний** под полем (можно tap-to-fill):

1. «Женский деловой портрет в светлой студии, реализм, мягкий свет»
2. «Уютный домик в зимнем лесу, вечер, тёплый свет из окон»
3. «Современный город ночью, неоновые вывески, кинематографичный стиль»
4. «Рекламное фото чашки кофе на деревянном столе, мягкий утренний свет»

**Цель:** научить писать **понятные** описания без слова «промпт».

---

Вкладка **«Создать»** поддержит **два режима** (целевое состояние):

| Режим | Ввод | Результат |
|-------|------|-----------|
| **1. По описанию** | Только **описание / идея** | **Одно** изображение |
| **2. Фото + описание** | **Фото** + **описание** | **Одно новое** изображение (не фотосессия) |

**Отличие от «Фотосессии»:**

| | **«Создать»** (фото + описание) | **«Фотосессии»** |
|---|--------------------------------|------------------|
| Результат | **1** изображение | **Несколько** (обычно **3**) в **выбранном стиле** |
| Сценарий | Своя идея + своё фото | Готовый **стиль** из каталога + своё фото |

### Реализовано сейчас

**Режим 1 — по описанию (полностью):**

- Поле **описания**, placeholder «Например: портрет в деловом стиле»
- **Быстрые идеи** (chips) — **план:** по категориям (см. выше)
- Блок **«Как получить хороший результат»** — **план:** расширить + 4 примера описаний
- **План:** строка баланса / *«3 бесплатные генерации»*
- **Модальное ожидание** при генерации (~60 с) — см. § **2.2**
- Кнопка **«Создать изображение»** → `POST /generate` (**текст** → одно изображение); **план:** блокировка без баланса после исчерпания free
- После успеха: элемент **сверху** в **Галерею**; **«Открыть в Галерее»**; fallback-preview
- При ошибке: SnackBar **«Не удалось создать изображение…»**; технические детали в UI **не показываются**

**Режим 2 — только UI-каркас (backend позже):**

- Блок **«Фото для образа»**: **«Добавить фото»** (`image_picker`), **preview**, **«Убрать фото»**
- Фото хранится **только локально** на экране; **не** уходит на backend, **не** в Gallery / `shared_preferences`
- Пояснение: *«Позже здесь можно будет создать одно изображение на основе вашего фото и описания»*
- При «Создать» с выбранным фото: SnackBar *«Создание по фото будет добавлено позже…»*, затем **генерация по описанию** как раньше
- **Контекстная помощь** обновлена: шаг **«Фото для образа»**

### Фото + описание → backend (план)

1. Пользователь **загружает фото** и пишет **описание / идею**.
2. Backend генерирует **одно** изображение → **Галерея** / `generations`.

**Примеры описаний:** «Сделай меня в деловом костюме», «Зимний портрет», «Аватарка для соцсетей».

**Статус backend:** endpoint и подключение Flutter — **не реализовано**; см. [roadmap.md](roadmap.md).
